# Appendix E — Adversarial review (pre-commit, 2026-06-09)

> Four independent reviewers (spec-vs-writer byte audit, regression audit of every modified file,
> security review, completeness critic) ran over the uncommitted Phase 39 implementation before
> the first commit. Every blocker/major was fixed before landing; minors/infos were fixed or are
> recorded in the phase doc. Verbatim verdicts + findings below (tool-generated, preserved for
> the record).

## Reviewer: spec

**Verdict:** The fx_2_0 writer, CTAB reader, and effect builder are byte-faithful to docs/fx2-binary-format.md in every structural respect I could test: header/pool conventions (offsets relative to 8, zeroed first dword, 4-aligned strings and pool size), typedef layouts including the dword5=columns/dword6=rows order and the square-matrix F1 guard, value-blob sizing (R×C×max(1,E) for numerics, state lists for samplers, object indices), parameter/technique/pass/state record sequences, small/large object records with the 4·⌈len/4⌉ advance, and the back-reference triples. I compiled a 2-technique/3-pass/2-sampler/render-state probe through the real CLI pipeline and decoded the output with an independent script: state_index correctly offsets past render states (2,3 / 1 / 0), sampler back-refs are (0xFFFFFFFF, paramIdx, stateIdx) with element −1, object indices are assigned textures→sampler-Texture-states→shaders starting at 1 with object_count including reserved 0, all dictionary iterations are deterministically ordered, render/sampler state values land as correct D3D9 enums (AlphaBlendEnable→1, SRCALPHA→5, CULLNONE→1, LINEAR→2), and there are zero trailing bytes. CtabReader's field widths/offsets match §11 exactly (u16 RegisterSet/Index/Count, Rows@+4/Columns@+6, not conflated with the §6 swap) and its bounds checks are equal-or-stricter than MojoShader's. However, the hunt the prompt requested on the int/bool default path found a real bug: an INT- or BOOL-typed global with an initializer gets its default written as IEEE-754 float bits, where the fx_2_0 convention (spec §1, vkd3d's fx.c writer, FNA's SetValue(int)) is raw integer dwords — empirically confirmed: `int Count = 7;` produces value blob 0x40E00000 (7.0f), which MojoShader uploads as (float)1088421888 instead of 7.0. That is wrong rendering on plausible input and is invisible to all 711 tests (every default-value test uses Type=FLOAT, and Fx2BinaryValidator checks blob bounds, never content-vs-type). Remaining findings are hardening/hygiene gaps.

### [major] INT/BOOL parameter defaults are written as float bits instead of raw integer dwords — wrong constant values under FNA

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Core\Fx2EffectWriter.cs`

Fx2EffectWriter.cs:184–192 writes every numeric default via pool.Write(float), regardless of param.Type. The fx_2_0 value-blob convention for INT/BOOL params is raw integer dwords (docs/fx2-binary-format.md §1 lines 37–38: 'Bools are stored as dwords 0/1 (vkd3d fx.c:1830 normalizes with !!value). Ints are dwords.'; §7.1: at draw MojoShader's copy_parameter_data converts INT/BOOL per-component from valuesI, and FNA's EffectParameter.SetValue(int)/GetValueInt32 use raw ints in the same memory). But vkd3d 1.17's CTAB stores defaults as a float register image even for INT/BOOL types — empirically verified by compiling `int Count = 7; bool Flag = true;` through the real pipeline (ShadowDuskCLI /Profile:FNA): the CTAB DefaultValue for Count is 0x40E00000 (7.0f) and for Flag 0x4F800000 ((float)0xFFFFFFFF). CtabReader.cs:137–147 reads those floats (semantically correct for CTAB), Fx2EffectBuilder.cs:144 passes them through, and the writer emits the float BITS into the blob: the emitted .fxb contains 0x40E00000 for Count and 0x4F800000 for Flag (decoded from the probe output). At FNA load, MojoShader stores valuesI[0]=1088421888 for Count and, because Count's register set is FLOAT4 ('Sometimes int/bool parameters get thrown into float registers'), uploads (float)1088421888 ≈ 1.09e9 to c1 where the shader expects 7.0 — `return Tint * Count;` renders garbage until game code calls SetValue; EffectParameter.GetValueInt32 also returns 1088421888. The missing piece is a type-aware conversion (float→int dword, bool normalized to 0/1) in the builder or writer before emission. No test covers this: every DefaultValue test in Fx2EffectBuilderTests/Fx2EffectWriterTests/CtabReaderTests uses Type=FLOAT constants, and Fx2BinaryValidator validates only blob size/bounds, never content vs declared type. Not in the spec's §15 ambiguity list either — F2 covers matrices only; §7.1 should gain an int/bool-encoding note when fixed.

### [minor] Sampler array element count silently dropped — `sampler ss[2]` emits a non-array sampler typedef

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Compiler\Internal\Fx2EffectBuilder.cs`

Fx2EffectBuilder.cs:193–199 constructs sampler parameters from the CTAB constant but never sets Elements (record default 0), unlike the numeric path at line 143 which propagates c.Elements. A shader declaring `sampler ss[2] : register(s0);` and sampling both elements yields a CTAB constant with Elements=2/RegisterCount=2, but the emitted fx_2_0 typedef says element_count=0 (not an array). MojoShader still binds sampler registers from the CTAB so rendering may survive, but the effect's parameter shape is wrong (FNA EffectParameter.Elements empty vs fxc's 2), and the single shared state list diverges from fxc's per-element layout (§7.2: one object index per element). Because the builder masks Elements to 0, Fx2EffectWriter.Validate's `object parameter is an array — not supported` check (Fx2EffectWriter.cs:402–403) can never fire for samplers — the unsupported input is silently reshaped instead of failing loudly (violates the fail-loudly posture used everywhere else, e.g. struct globals → SD0303). Reject sampler-array CTAB constants explicitly or propagate Elements.

### [minor] Fx2EffectWriter.Validate gaps for direct (non-builder) callers: negative Elements emits a memory-corrupting typedef; non-ASCII names silently mangled

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Core\Fx2EffectWriter.cs`

Fx2EffectWriter is public API and its header comment promises 'every invariant MojoShader relies on must be enforced here', but Validate (Fx2EffectWriter.cs:363–490) misses two: (1) Elements is never checked for negative values — a numeric param with Elements=-1 passes validation (DefaultValue check at :393 uses Math.Max(1,Elements) and only fires when DefaultValue is non-null), then line 181 writes (uint)param.Elements = 0xFFFFFFFF as the typedef element_count while the blob (line 184, Math.Max(1,...)) holds one element; MojoShader allocates value_count = 4·R·0xFFFFFFFF and reads R×E rows from the pool — exactly the §1 'silent memory corruption' the validator exists to prevent. (2) AddString (line 514) uses Encoding.ASCII.GetBytes, which converts non-ASCII chars to '?'; a parameter name containing non-ASCII then no longer strcmp-matches the CTAB constant name in the embedded bytecode — §13.10 release-mode UB in MojoShader. Neither is reachable through Fx2EffectBuilder (CTAB Elements is a u16 normalized via Math.Max(1,·); vkd3d rejects non-ASCII identifiers), so this is hardening for the public-API surface only, not a pipeline bug.

### [minor] Duplicate render-state keys in a pass throw unhandled ArgumentException instead of a ShaderError

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Compiler\Internal\CompilationPipeline.cs`

CompilationPipeline.cs:564–565 (RunFnaAsync) does `pass.RenderStates.ToDictionary(rs => rs.Key, rs => rs.Value, StringComparer.OrdinalIgnoreCase)`. A pass that assigns the same render state twice — `pass P0 { CullMode = CW; CullMode = NONE; PixelShader = compile ps_3_0 PS(); }`, which fxc accepts (last assignment wins) — throws System.ArgumentException out of ToDictionary instead of returning a Result error with file/line, violating constraint 5 (fail loudly with diagnostics) and the no-exception-as-control-flow convention. This mirrors the identical pre-existing pattern in the MGFX path (CompilationPipeline.cs:344–345), so it is not an FNA regression, but the new code re-instantiates the bug rather than fixing or routing around it (e.g. last-wins overwrite plus a warning, matching fxc).

### [minor] vkd3d 1.17 ignores #line directives — FNA-path diagnostic positions reference flattened text, plus 'fixme' noise on stderr for every compile

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Compiler\Internal\CompilationPipeline.cs`

Every FNA compile through the real pipeline prints `vkd3d:NNNN:fixme:preproc_yyparse #line directive.` to stderr (observed once per compiled stage on successful probe compiles): the Preprocessor's flatten step (CompilationPipeline.cs:494–500) embeds #line markers that vkd3d 1.17's preprocessor parses but does not implement. Consequences: (a) vkd3d error line numbers for the FNA target refer to the flattened, macro-prelude-prefixed text rather than the user's source file/line — a constraint-5 ('exactly as the underlying compiler emitted them') fidelity gap adjacent to, but distinct from, the empty-messages-blob limitation already recorded in plan/PHASE-39-fna-fx2-output-target.md; (b) raw native 'fixme' noise leaks to the CLI's stderr on success, which MGCB-style tooling may treat as diagnostic output. Worth a line in the phase doc's Known limitations and a stderr filter or vkd3d log-level adjustment.

### [info] Shared entry points are recompiled and embedded once per referencing pass (no object sharing)

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Compiler\Internal\CompilationPipeline.cs`

CompilationPipeline.cs:529–562 assigns a fresh shader index (and Fx2EffectWriter.cs:92–103 a fresh object + duplicate large record) for every pass that references an entry point, even when identical. Empirically: a probe where technique T2 reuses PS1 from T1 produced two byte-identical 256-byte embedded copies (large records #1 and #3), each compiled separately by vkd3d and each recompiled by MojoShader at load. fxc shares one object per compiled function across passes. Fully spec-legal and deterministic — cost is file size and redundant compiles only. Worth noting as a future dedup opportunity (key the shader list on (entryPoint, profile, stage)).

### [info] The production writer never enforces the CTAB-name ⊆ parameter-name hard requirement (§13.10) — only the test-side validator does

*File:* `C:\git\ShadowDusk\src\ShadowDusk.Core\Fx2EffectWriter.cs`

Spec §13.10: every CTAB constant name in every embedded shader must exactly match an effect parameter name; a miss is params[-1] memory corruption in release-mode MojoShader. Today this holds only by construction in Fx2EffectBuilder (parameters are generated from the union of all CTABs) and is checked only by the test-side Fx2BinaryValidator (Fx2BinaryValidator.cs:752–761), never in the production path. Fx2EffectWriter.Validate (Fx2EffectWriter.cs:363–490) already has the shader bytecode (effect.Shaders) and CtabReader lives in the same assembly, so a cheap cross-check (parse each blob's CTAB, assert every constant name ∈ parameter names) would convert any future builder regression — or any direct-API misuse — from silent FNA memory corruption into a loud SD0302. Defense-in-depth suggestion; no current input violates it.

## Reviewer: security

**Verdict:** The new binary-parsing surface (CtabReader) is memory-safe and non-throwing under adversarial input — its offset arithmetic is promoted to long at every dangerous step, counts are double-bounded, and the comment-walk and constant loop both terminate and stay in-bounds. The fx_2_0 writer and the test-side validator are likewise robustly bounds-checked, and the FNA pipeline does no temp-file or attacker-path I/O. I found ONE concretely attacker-reachable defect: a quadratic-time (O(n^2)) blowup in Sm3StageReservationRewriter that runs synchronously on untrusted .fx source and is NOT interruptible by the pipeline's CancellationToken, giving an unbounded compile-time hang. The remaining items are defense-in-depth gaps not reachable from a vkd3d-produced CTAB. No output-corruption or FNA-crash path is reachable from untrusted source.

### [major] Quadratic-time DoS in Sm3StageReservationRewriter on untrusted .fx source (synchronous, not interruptible by CancellationToken)

*File:* `src/ShadowDusk.HLSL/Sm3StageReservationRewriter.cs`

Rewrite() (line 25) scans the full preprocessed HLSL char-by-char. When it hits ':' it calls TryMatchStageReservation (line 73). On a candidate that matches ': register ( vs|ps ,' the body loop at lines 154-173 scans forward to the matching ')'. If no ')' is found it returns false (line 175) AFTER scanning to EOF, and the outer loop then only advances i by 1 (lines 96-97, the false-match fall-through) — so the expensive forward scan is re-run at the very next ':'. An attacker-supplied .fx whose body is the literal token ':register(vs,' repeated (no closing paren anywhere; '(' inside each repeat keeps depth>0 so every scan runs to EOF) makes each of the ~n/13 colons trigger an O(n) scan => O(n^2) total. Concrete trigger: a valid one-technique/one-pass effect (so FxPreParser/Preprocessor succeed and the rewriter is reached at CompilationPipeline.cs:536/551) followed by ~5 MB of ':register(vs,' junk preserved verbatim into preprocessed.Text => ~2e12 char comparisons => multi-minute+ hang. Crucially Rewrite is synchronous and takes NO CancellationToken (line 25), and the call sites at CompilationPipeline.cs:536 and :551 do not pass the token, so the pipeline's compile timeout (the documented mitigation for 'compiler hangs') cannot preempt it — the calling thread is pinned on managed CPU before any cancellation-aware await. This is the only new managed scanner over untrusted source added in Phase 39; the existing tests (Sm3StageReservationRewriterTests.cs) cover correctness only, no pathological-input/timing case. Loop always terminates (every branch advances i>=1) so it is not an infinite loop — purely a complexity DoS. Fix: on a failed stage-reservation match advance i past the consumed 'register(' prefix (or memoize/limit the body scan), and thread the CancellationToken into Rewrite with periodic ThrowIfCancellationRequested.

### [minor] Fx2EffectWriter dword-count int overflow with validation bypass when DefaultValue is null (defense-in-depth gap in the 'never emit a corrupting binary' promise)

*File:* `src/ShadowDusk.Core/Fx2EffectWriter.cs`

Line 184 computes 'int dwordCount = param.Rows * param.Columns * elements' and line 393 computes the parallel 'int expected = ...' — both in unchecked int. Rows/Columns originate from CtabReader as raw u16 (CtabReader.cs:124-125, no upper bound; Class/Type validated but dimensions are not), and Elements up to 65535. If Rows*Columns*Elements exceeds int.MaxValue the product wraps; e.g. a class-1 (vector) param with Rows=65535,Columns=65535 passes Validate (lines 383-385 only require >0 and square-for-matrix) and yields a negative dwordCount, so the value-blob loop at line 186 writes ZERO dwords while the emitted typedef (lines 179-181) advertises huge dims — MojoShader/FNA would then read far past the blob (out-of-bounds read at FNA load). The expected-vs-actual guard at line 394 is skipped entirely when param.DefaultValue is null, so nothing catches the mismatch. NOTE ON REACHABILITY: this is NOT reachable from the documented threat model (untrusted .fx -> vkd3d -> CTAB), because vkd3d reflects real HLSL types (Rows/Columns <= 4) and the SM3 constant-register file caps array Elements at ~256, so the product is always tiny. It is a latent gap should the CTAB ever be malformed/fuzzed or sourced differently. Fix: cap Rows/Columns (and Elements) to a sane max in CtabReader and/or Fx2EffectWriter.Validate, and validate dwordCount in checked arithmetic regardless of whether DefaultValue is present.

### [minor] Fx2EffectBuilder silently keeps first SamplerInfo on duplicate sampler names (TryAdd) — masks malformed input rather than failing loudly

*File:* `src/ShadowDusk.Compiler/Internal/Fx2EffectBuilder.cs`

Line 124-126 builds samplerInfoByName with TryAdd, so if two SamplerInfo entries share a name the second (and its sampler_state block / texture reference) is silently dropped. This correctly avoids a dictionary-poisoning crash (good — no throw), but it means a .fx with a duplicated sampler declaration produces an effect bound to the first declaration's states with no diagnostic, contrary to constraint 5 (fail loudly). Reachability is low: HLSL forbids two globals of the same name, so vkd3d would reject the source before this point; and even a same-name numeric+texture collision is caught downstream by Fx2EffectWriter.Validate's duplicate-name check (Fx2EffectWriter.cs:376-377) which fails cleanly. No memory-safety impact. Hygiene only.

### [info] CtabReader / Fx2 pipeline binary-parsing defenses are sound (positive confirmation)

*File:* `src/ShadowDusk.Core/Reflection/CtabReader.cs`

Verified the items the review flagged as risk areas are correctly handled: (1) offset arithmetic that could overflow is promoted to long before comparison — payloadEnd (line 56), constantInfoOfs+constantCount*20 (line 95), typeInfoOfs+16 (line 119), defaultOfs+columns*4 (line 139); (2) every '(int)' cast before a Slice (lines 106,122-123,145,175,179) is guarded by a prior bounds check, so no negative/out-of-range Slice; (3) the leading comment-walk advances pos by 4 even when commentDwords==0 (lines 54-68) so no infinite loop, and a first-instruction/high-bit token breaks out; (4) constantCount is capped at 1,000,000 (line 93) and further bounded by the record-fit check at line 95 (region<=131KB => effective cap ~6.5K), so the List<CtabConstant> capacity at line 103 is not an allocation bomb; (5) TryReadString rejects unterminated strings (line 177) and out-of-region offsets (line 173); (6) default-value reads are bounded so a large 'columns' simply skips (defEnd>region.Length) rather than over-reading.
